Declarative Macros

Declarative macros (macro_rules!), as a crucial component of Rust’s syntax extension system, are not core knowledge for beginners but offerirreplaceable value in lightweight code generation and syntax simplification scenarios. This tutorial starts with basic concepts, gradually delves into the matching rules, repetition logic, and scope characteristics of declarative macros, and breaks down difficulties with practical examples. It finally realizes knowledge application through a hands-on project, helping developers systematically master this advanced skill.

Chapter 1: Fundamentals of Declarative Macros: Nature and Introduction

1.1 Core Definition of Declarative Macros

Declarative macros are syntax extension tools provided by Rust based on “pattern matching-code expansion”. Defined via the macro_rules! keyword, they essentially match, replace, and generate code snippets (TokenTree) at compile time, ultimately outputting code that conforms to Rust syntax. Unlike regular functions and generics, macros do not participate in runtime logic and only complete code transformation during the compilation phase. Thus, they can achieve syntax abstraction scenarios that generics and functions cannot cover (such as generating vec![] literals).

Core advantages: Lightweight with no additional compilation dependencies, supporting syntax snippet-level reuse; Core limitations: Syntax is independent of Rust’s main syntax, error messages are obscure, and complex logic processing is not supported.

1.2 Minimal Viable Macro: Hello Macro Example

Let’s start with the simplest macro to understand its basic structure. The following example implements a macro that prints fixed information:


// Define a declarative macro
macro_rules! hello_macro {
    // Matching rule: no-argument pattern
    () => {
        // Expanded code
        println!("Hello, Rust Declarative Macro!");
    };
}

fn main() {
    // Call the macro (parentheses are optional but recommended for consistency)
    hello_macro!(); // Expands to println!(...) at compile time
}

Code Explanation:

  • macro_rules! macro_name { ... }: The fixed definition format for declarative macros.

  • () => { ... }: A matching arm. () is the pattern (representing a no-argument call), and the code block after => is expanded when the pattern matches successfully.

  • Macro calls require !: Used to distinguish macros from functions/methods, informing the compiler to perform macro expansion here.

1.3 Macro Matching Patterns and Metavariables

The core capability of macros is to match different syntax snippets, capture matched content via metavariables, and reuse it. Metavariables must specify a “fragment specifier” (such as expression, identifier, statement). Common fragment specifiers are as follows:

Fragment SpecifierDescriptionExample
exprAny expression1+2, foo(), x>3
identIdentifier (variable name, function name, etc.)x, hello_world, User
stmtSingle statementlet a = 5;, foo();
blockCode block{ let a = 5; a+1; }
tyTypei32, String, &[u8]
Example: Implement a macro that captures an expression and prints its content and result:
 

macro_rules! print_expr {
    // Match any expression, capture with $expr:expr, where $expr is the metavariable name
    ($expr:expr) => {
        println!("Expression `{} = {}`", stringify!($expr), $expr);
    };
}

fn main() {
    let x = 10;
    print_expr!(x + 5); // Expands to println!("Expression `x + 5 = {}`", x + 5);
    print_expr!(2 * 3 + 4); // Expands to the corresponding print statement
}

Key Knowledge: stringify! is a built-in Rust macro that converts the syntax snippet corresponding to a metavariable into a string literal, commonly used for debugging or printing the expression itself.

Chapter 2: Advanced Declarative Macros: Repetition, Multiple Arms, and Scope

2.1 Repeaters: Handling Variable-Length Arguments

When needing to match a variable number of syntax snippets (such as multiple elements in vec![1,2,3]), macro repeater syntax is required. The core format of a repeater is $()sep*, where:

  • $(): Wraps the pattern to be repeated.

  • sep: Optional separator (such as comma, semicolon) used to separate multiple repeated snippets.

  • *: Repetition quantifier, supporting * (zero or more times), + (one or more times), and ? (zero or one time).

Example 1: Implement a simplified vec! macro (supports only elements of the same type):

#![allow(unused)]
fn main() {

macro_rules! my_vec {
    // Match empty call: my_vec![]
    () => {
        Vec::new()
    };
    // Match multiple expressions separated by commas, + means at least one element
    ($($elem:expr),+) => {
        {
            let mut vec = Vec::new();
            // Repeat expansion: call push for each captured $elem
            $(vec.push($elem);)+
            vec
        }
    };
}
}

Call Verification:


fn main() {
    let v1 = my_vec![];
    let v2 = my_vec![1, 2, 3, 4];
    println!("v1: {:?}, v2: {:?}", v1, v2); // Output: v1: [], v2: [1,2,3,4]
}

Example 2: Support trailing commas (more in line with Rust syntax conventions) by adding a new arm:

#![allow(unused)]
fn main() {

macro_rules! my_vec {
    () => { Vec::new() };
    ($($elem:expr),+) => { my_vec!($($elem),+,) }; // Forward to the trailing comma arm
    ($($elem:expr),+,) => { // Match cases with a trailing comma
        {
            let mut vec = Vec::new();
            $(vec.push($elem);)+
            vec
        }
    };
}
}

Now both my_vec![1,2,3,] and my_vec![1,2,3] can be called normally, with better compatibility.

2.2 Multiple Arms Matching: Handling Different Call Scenarios

Declarative macros support multiple matching arms. The compiler matches arms in order; if a match is successful, it expands the corresponding code. If no match is found, it reports a compilation error. Multiple arms can handle scenarios with different parameter types and call formats.

Example: Implement a calc! macro that supports addition and multiplication operations, and handles single arguments (returns the argument itself):


macro_rules! calc {
    // Arm 1: Single argument, return directly
    ($x:expr) => { $x };
    // Arm 2: Addition operation, matches "a + b" format
    ($a:expr + $b:expr) => { $a + $b };
    // Arm 3: Multiplication operation, matches "a * b" format
    ($a:expr * $b:expr) => { $a * $b };
}

fn main() {
    assert_eq!(calc!(5), 5);
    assert_eq!(calc!(3 + 4), 7);
    assert_eq!(calc!(2 * 6), 12);
    // calc!(3 - 2); // Compilation error: no matching arm
}

Note: The order of arms affects matching results. More specific arms should be placed first (e.g., operation arms are more specific than single-argument arms and should be defined first).

2.3 Macro Scope and Hygiene

Rust declarative macros are hygienic, meaning variables and identifiers expanded from macros do not conflict with the scope of the call site. This is one of the core advantages of Rust macros over C/C++ macros.

Example: Variables defined inside a macro do not pollute the outer scope:


macro_rules! create_var {
    () => {
        let x = 10;
        println!("x inside macro: {}", x);
    };
}

fn main() {
    create_var!(); // x is defined after expansion, valid only in the local scope of the macro expansion
    // println!("x outside macro: {}", x); // Compilation error: x is not defined
}

When needing to reference identifiers from the outer scope in a macro or expose identifiers defined in the macro to the outer scope, note:

  • Macros cannot capture variables from the outer scope by default unless the variables are global or passed as arguments.

  • The $crate:: prefix can reference top-level identifiers of the crate where the macro resides, ensuring correct paths for cross-module calls (suitable for library macros).

Chapter 3: Hands-On Project: Building a Lightweight Logging Library

3.1 Project Requirement Analysis

In this hands-on project, we will develop a lightweight logging library named mini_log using declarative macros. It will have the following features:

  1. Support 4 log levels: INFO, WARN, ERROR, DEBUG.

  2. Log output format: [Level] [Time] Message Content (time is simplified to the current timestamp in seconds, using std::time::SystemTime).

  3. Support variable-argument messages (e.g., log_info!("User {} logged in, IP:{}", username, ip)).

  4. Support enabling/disabling DEBUG level logs (controlled at compile time to avoid redundant DEBUG logs in production environments).

3.2 Core Function Implementation

3.2.1 Basic Log Macro Definition

First, implement the general log formatting logic. We will distinguish different log levels via macro arms:

#![allow(unused)]
fn main() {

use std::time::{SystemTime, UNIX_EPOCH};

// Helper function: Get current timestamp in seconds
fn get_timestamp() -> u64 {
    SystemTime::now()
        .duration_since(UNIX_EPOCH)
        .unwrap()
        .as_secs()
}

// General log macro: Accepts level and message arguments
macro_rules! log {
    ($level:ident, $($arg:tt)+) => {
        println!(
            "[{}] [{}] {}",
            stringify!($level).to_uppercase(),
            get_timestamp(),
            format!($($arg)+)
        );
    };
}

// Level-specific log macros: Wrap the log! macro
macro_rules! log_info {
    ($($arg:tt)+) => { log!(INFO, $($arg)+); };
}

macro_rules! log_warn {
    ($($arg:tt)+) => { log!(WARN, $($arg)+); };
}

macro_rules! log_error {
    ($($arg:tt)+) => { log!(ERROR, $($arg)+); };
}

// DEBUG log: Controlled by compile-time cfg
#[cfg(debug_assertions)]
macro_rules! log_debug {
    ($($arg:tt)+) => { log!(DEBUG, $($arg)+); };
}

#[cfg(not(debug_assertions))]
macro_rules! log_debug {
    ($($arg:tt)+) => { /* Empty expansion in production, no performance overhead */ };
}
}

Code Explanation:

  • $($arg:tt)+: tt (TokenTree) is a fragment specifier that matches any token (including identifiers, operators, parentheses, etc.). It is suitable for handling variable-argument formatted strings (like the argument format of println!).

  • cfg(debug_assertions): A built-in Rust conditional compilation flag. It is enabled by default in cargo build (debug mode) and disabled in cargo build --release (release mode), realizing compile-time switching of DEBUG logs.

  • format!($($arg)+): Formats variable arguments into a string, then passes it to the log output statement. This maintains the same usage as println!.

Chapter 4: Common Pitfalls and Best Practices of Declarative Macros

4.1 Common Pitfalls and Avoidance Guides

4.1.1 Expression Re-evaluation Issue

Macro expansion directly replaces code. If a metavariable is an expression with side effects (e.g., function call, increment operation), it will cause re-evaluation:


macro_rules! double {
    ($x:expr) => { $x + $x };
}

fn main() {
    let mut a = 1;
    let b = double!(a += 1); // Expands to (a +=1) + (a +=1), a becomes 3, b=2+3=5
    println!("a: {}, b: {}", a, b);
}

Avoidance Solution: Bind the metavariable to a temporary variable to prevent re-evaluation:

#![allow(unused)]
fn main() {

macro_rules! double {
    ($x:expr) => {
        {
            let temp = $x;
            temp + temp
        }
    };
}
}

4.1.2 Obscure Error Messages Issue

When macro matching fails or syntax errors occur after expansion, compiler error messages often point to the macro call site rather than the problem in the macro definition. Avoidance Solutions:

  • Split complex macros into multiple simple macros to reduce the complexity of matching logic.

  • Use the compile_error! built-in macro to add custom error messages for better readability:

#![allow(unused)]
fn main() {

macro_rules! only_u32 {
    ($x:expr) => {
        match $x {
            _: u32 => $x,
            _ => compile_error!("Only u32 type arguments are supported"),
        }
    };
}
}

4.1.3 Cross-Module Call Path Issue

When a macro defined in a library references functions/types from its own crate, using relative paths directly may cause cross-module call failures. Avoidance Solution: Use the $crate:: prefix to reference top-level paths:

#![allow(unused)]
fn main() {

// Library crate: mini_log
pub mod utils {
    pub fn get_timestamp() -> u64 { /* ... */ }
}

pub macro_rules! log_info {
    ($($arg:tt)+) => {
        println!(
            "[INFO] [{}] {}",
            $crate::utils::get_timestamp(), // Safe cross-module reference
            format!($($arg)+)
        );
    };
}
}

4.2 Best Practices

  1. Prefer generics/functions over macros: Use declarative macros only when generics and functions cannot implement syntax expansion or code simplification.

  2. Keep macros concise: Each macro is responsible for a single logic; split complex logic into helper functions or multiple macros.

  3. Add documentation comments: Write detailed call examples and parameter descriptions for macros to make up for the poor readability of macro syntax.

  4. Test thoroughly: Cover different call scenarios (e.g., empty arguments, trailing commas, expressions with side effects) to avoid hidden issues.

  5. Use the tt fragment specifier cautiously: tt has a broad matching range and may cause unexpected matches. Prefer more specific fragment specifiers (e.g., expr, ident).

Rust Mastery: Conquering Procedural Macros

Procedural macros are a powerful feature in Rust that enable metaprogramming—writing code that generates code. Unlike declarative macros (e.g., macro_rules!), which operate on token streams using pattern matching, procedural macros are compiled Rust functions that take a token stream as input, process it, and emit a new token stream as output. This capability unlocks advanced use cases such as automatic trait implementation, data structure serialization, and domain-specific language (DSL) creation.

This tutorial aims to demystify procedural macros, covering their types, core mechanics, implementation details, and best practices. Each section includes concise, runnable code examples to reinforce concepts, followed by practice exercises to test your understanding. The final chapter guides you through a practical project that integrates multiple procedural macro techniques, helping you apply what you’ve learned to real-world scenarios.

Prerequisites: Familiarity with Rust fundamentals (ownership, traits, generics), basic macro syntax (macro_rules!), and Rust’s crate system. You should also have Rust 1.56+ installed (as procedural macro APIs have stabilized in recent versions).

Chapter 1: Introduction to Procedural Macros

1.1 What Are Procedural Macros?

Procedural macros are Rust functions annotated with special attributes that allow them to manipulate Rust code at compile time. They differ from declarative macros in two key ways: first, they use Rust’s full type system and logic capabilities (no limited pattern matching), and second, they operate on the TokenStream type (from the proc_macro crate), which represents the raw syntax of Rust code.

When the Rust compiler encounters a procedural macro, it invokes the corresponding function, passes the tokens of the code the macro is applied to, and replaces the macro invocation with the token stream returned by the function. This compilation-time processing ensures that generated code is type-checked and optimized alongside handwritten code.

1.2 Types of Procedural Macros

Rust defines three categories of procedural macros, each with a distinct purpose and application syntax:

  1. Function-like Macros: Invoked like regular functions (e.g., derive_json!(MyStruct)). They take a token stream as input and return a token stream, with no restrictions on the input syntax (beyond being valid Rust tokens).

  2. Derive Macros: Applied via the #[derive(MacroName)] attribute on structs, enums, or unions. They generate trait implementations for the annotated type. Examples include #[derive(Debug)] (built-in) and #[derive(Serialize)] (from serde).

  3. Attribute Macros: Applied as custom attributes (e.g., #[route("/api/users")]) on functions, structs, enums, modules, or traits. They can modify the annotated item or generate additional code alongside it.

1.3 Procedural Macro Crates

Procedural macros must be defined in a separate crate with the proc-macro crate type. This is a compiler requirement to isolate macro code (which runs at compile time) from the main application code (which runs at runtime). To create a procedural macro crate, add the following to Cargo.toml:

[package]
name = "my-proc-macros"
version = "0.1.0"
edition = "2021"

[lib]
proc-macro = true

[dependencies]
proc-macro2 = "1.0"  # Wrapper for proc_macro with additional features
quote = "1.0"        # Converts Rust syntax trees to TokenStream
syn = "2.0"          # Parses TokenStream into Rust syntax trees

Key dependencies:

  • proc-macro: The standard library crate providing core procedural macro APIs (only available in proc-macro crates).

  • proc-macro2: A drop-in replacement for proc-macro that works in non-proc-macro crates, enabling code sharing.

  • syn: A parser that converts TokenStream into a structured syntax tree (AST) for easy manipulation.

  • quote: A crate that converts Rust AST nodes back into TokenStream, simplifying code generation.

1.4 Hello, Procedural Macro!

Let’s implement a simple function-like procedural macro that generates a greeting function. First, create the proc-macro crate (as above), then add the following code to src/lib.rs:

#![allow(unused)]
fn main() {
// src/lib.rs
use proc_macro::TokenStream;
use quote::quote;
use syn::{parse_macro_input, LitStr};

#[proc_macro]
pub fn greet(input: TokenStream) -> TokenStream {
    // Parse the input token stream as a string literal
    let name = parse_macro_input!(input as LitStr);
    
    // Generate Rust code: a function that prints a greeting
    let expanded = quote! {
        fn say_hello() {
            println!("Hello, {}!", #name);
        }
    };
    
    // Convert the generated syntax tree back to TokenStream
    TokenStream::from(expanded)
}

}

To use this macro in a separate binary crate, add the proc-macro crate as a dependency in Cargo.toml:

[dependencies]
my-proc-macros = { path = "../my-proc-macros" }

Then invoke the macro in src/main.rs:

// src/main.rs
use my_proc_macros::greet;

greet!("Alice");

fn main() {
    say_hello(); // Prints "Hello, Alice!"
}

When compiled, the greet!("Alice") invocation is replaced with the say_hello function generated by the macro. The compiler type-checks the generated code, so if you pass an invalid input (e.g., a number instead of a string), syn will return a compile error.

Chapter 2: Derive Macros

2.1 Fundamentals of Derive Macros

Derive macros are the most common type of procedural macro. They allow users to automatically implement traits for their types by adding #[derive(TraitName)]. Under the hood, the derive macro parses the annotated type (struct/enum/union), generates the trait implementation code, and injects it into the compilation process.

To define a derive macro, use the #[proc_macro_derive(TraitName)] attribute on a function. The function takes a TokenStream representing the annotated type, parses it with syn, generates the trait implementation with quote, and returns the generated TokenStream.

2.2 Implementing a Custom Derive Macro

Let’s create a derive macro MyDebug that generates a simple debug implementation (similar to the built-in Debug trait). First, add the trait definition (in a separate crate or the same binary crate) and the derive macro in the proc-macro crate.

Step 1: Define the MyDebug trait in a shared crate (e.g., my-traits):

#![allow(unused)]
fn main() {
// my-traits/src/lib.rs
pub trait MyDebug {
    fn my_debug(&self) -> String;
}

}

Step 2: Implement the derive macro in my-proc-macros/src/lib.rs:

#![allow(unused)]
fn main() {
// src/lib.rs
use proc_macro::TokenStream;
use quote::quote;
use syn::{parse_macro_input, DeriveInput, Fields, Ident};

// Derive macro for MyDebug trait
#[proc_macro_derive(MyDebug)]
pub fn derive_my_debug(input: TokenStream) -> TokenStream {
    // Parse the input into a DeriveInput (represents a struct/enum/union)
    let input = parse_macro_input!(input as DeriveInput);
    
    // Extract the name of the type (e.g., "User" for struct User { ... })
    let type_name = input.ident;
    
    // Generate the trait implementation based on the type's fields
    let impl_block = match input.data {
        syn::Data::Struct(data) => match data.fields {
            Fields::Named(fields) => {
                // For structs with named fields (e.g., struct S { a: i32, b: String })
                let field_names: Vec<Ident> = fields.named.iter()
                    .map(|field| field.ident.as_ref().unwrap())
                    .collect();
                
                quote! {
                    impl MyDebug for #type_name {
                        fn my_debug(&self) -> String {
                            format!(
                                "{} {{ {} }}",
                                stringify!(#type_name),
                                #(
                                    format!("{}: {:?}", stringify!(#field_names), &self.#field_names)
                                ),*
                            )
                        }
                    }
                }
            }
            Fields::Unnamed(fields) => {
                // For structs with unnamed fields (e.g., struct S(i32, String))
                let field_indices: Vec<usize> = (0..fields.unnamed.len()).collect();
                
                quote! {
                    impl MyDebug for #type_name {
                        fn my_debug(&self) -> String {
                            format!(
                                "{}({})",
                                stringify!(#type_name),
                                #(
                                    format!("{:?}", &self.#field_indices)
                                ),*
                            )
                        }
                    }
                }
            }
            Fields::Unit => {
                // For unit structs (e.g., struct S;)
                quote! {
                    impl MyDebug for #type_name {
                        fn my_debug(&self) -> String {
                            stringify!(#type_name).to_string()
                        }
                    }
                }
            }
        },
        syn::Data::Enum(_) | syn::Data::Union(_) => {
            // For simplicity, we'll only support structs in this example
            panic!("MyDebug is only supported for structs");
        }
    };
    
    TokenStream::from(impl_block)
}

}

Step 3: Use the derive macro in a binary crate:

// src/main.rs

use my_traits::MyDebug as MyDebugTrait;

#[derive(MyDebug)]
struct User {
    name: String,
    age: u32,
}

#[derive(MyDebug)]

        },

Chapter 3: Attribute Macros

3.1 How Attribute Macros Work

Attribute macros modify or augment Rust items (functions, structs, modules, etc.) by applying custom attributes. Unlike derive macros, which only generate trait implementations, attribute macros can alter the annotated item itself or generate additional code (e.g., functions, constants) alongside it.

To define an attribute macro, use the #[proc_macro_attribute] attribute on a function. The function signature differs from other procedural macros: it takes two TokenStream arguments—one for the attribute’s arguments (if any) and one for the annotated item. It returns a TokenStream representing the modified item (and any generated code).

3.2 Implementing a Simple Attribute Macro

Let’s create an attribute macro #[log_execution] that adds logging before and after a function’s execution. The macro will print the function name when it starts and finishes.

#![allow(unused)]
fn main() {
// my-proc-macros/src/lib.rs
use proc_macro::TokenStream;
use quote::quote;
use syn::{parse_macro_input, ItemFn, LitStr};

#[proc_macro_attribute]
pub fn log_execution(attr: TokenStream, item: TokenStream) -> TokenStream {
    // Parse the attribute arguments (optional message)
    let msg = if attr.is_empty() {
        LitStr::new("Executing function", proc_macro2::Span::call_site())
    } else {
        parse_macro_input!(attr as LitStr)
    };
    
    // Parse the annotated function
    let mut func = parse_macro_input!(item as ItemFn);
    
    // Extract the function name
    let func_name = &func.sig.ident;
    
    // Modify the function body to add logging
    let original_body = func.block;
    func.block = syn::parse_quote!({
        println!("{}: {}", #msg, stringify!(#func_name));
        let result = #original_body;
        println!("Completed executing: {}", stringify!(#func_name));
        result
    });
    
    // Generate the modified function
    TokenStream::from(quote! { #func })
}

}

3.3 Attribute Macros with Complex Arguments

Attribute macros can accept structured arguments (e.g., key-value pairs) by parsing the attribute token stream into a custom AST. Let’s extend #[log_execution] to support a level argument (e.g., #[log_execution(level = "debug")]) that controls the logging level.

#![allow(unused)]
fn main() {
// my-proc-macros/src/lib.rs
use syn::{Meta, MetaNameValue, NestedMeta, Lit, LitStr};

#[proc_macro_attribute]
pub fn log_execution(attr: TokenStream, item: TokenStream) -> TokenStream {
    // Parse attribute arguments into a MetaList (e.g., log_execution(level = "debug", msg = "Start"))
    let meta_list = parse_macro_input!(attr as Meta);
    let mut level = LitStr::new("info", proc_macro2::Span::call_site());
    let mut msg = LitStr::new("Executing function", proc_macro2::Span::call_site());
    
    // Process each key-value pair in the attribute
    if let Meta::List(list) = meta_list {
        for nested_meta in list.nested {
            if let NestedMeta::Meta(Meta::NameValue(name_value)) = nested_meta {
                let key = name_value.path.is_ident("level");
                let msg_key = name_value.path.is_ident("msg");
                
                if key {
                    if let Lit::Str(lit_str) = name_value.lit {
                        level = lit_str;
                    }
                } else if msg_key {
                    if let Lit::Str(lit_str) = name_value.lit {
                        msg = lit_str;
                    }
                }
            }
        }
    }
    
    let func = parse_macro_input!(item as ItemFn);
    let func_name = &func.sig.ident;
    
    // Generate logging code based on the level
    let log_start = match level.value().as_str() {
        "debug" => quote! { eprintln!("[DEBUG] {}: {}", #msg, stringify!(#func_name)); },
        "warn" => quote! { eprintln!("[WARN] {}: {}", #msg, stringify!(#func_name)); },
        _ => quote! { println!("[INFO] {}: {}", #msg, stringify!(#func_name)); },
    };
    
    let log_end = quote! {
        println!("[{}] Completed: {}", #level, stringify!(#func_name));
    };
    
    // Modify the function body
    let original_body = func.block;
    let modified_body = syn::parse_quote!({
        #log_start
        let result = #original_body;
        #log_end
        result
    });
    
    let mut modified_func = func;
    modified_func.block = modified_body;
    
    TokenStream::from(quote! { #modified_func })
}

}

    TokenStream::from(quote! {
        #struct_def
        #type_alias
    })
}

Chapter 4: Function-Like Procedural Macros

4.1 Advanced Function-Like Macros

Function-like procedural macros are the most flexible type of procedural macro, as they can accept any valid Rust token stream as input. They are invoked like regular functions (with parentheses or braces) and can generate any Rust code, making them ideal for creating DSLs, code generators, and utility macros.

Unlike macro_rules!, function-like procedural macros can use Rust’s full logic to process input tokens. For example, you can iterate over a list of tokens, validate their structure, and generate code dynamically based on complex conditions.

4.2 Building a DSL with a Function-Like Macro

Let’s create a function-like macro html! that acts as a DSL for generating HTML strings. The macro will accept HTML-like syntax (e.g., html!(div { class: "container", p { "Hello World" } })) and generate Rust code that builds the corresponding HTML string.

#![allow(unused)]
fn main() {
// my-proc-macros/src/lib.rs
use proc_macro::TokenStream;
use quote::quote;
use syn::{parse_macro_input, Expr, ExprBlock, Ident, LitStr, Token, parse::Parser};

// Define the structure of an HTML element (tag, attributes, children)
#[derive(Debug)]
struct HtmlElement {
    tag: Ident,
    attrs: Vec<(Ident, LitStr)>,
    children: Vec<HtmlNode>,
}

#[derive(Debug)]
enum HtmlNode {
    Element(HtmlElement),
    Text(LitStr),
}

// Parser for the html! macro input
fn parse_html_input(input: TokenStream) -> HtmlElement {
    let parser = HtmlElement::parse;
    parser.parse(input).expect("Invalid HTML syntax in html! macro")
}

impl HtmlElement {
    // Parse an HTML element (e.g., div { class: "foo", p { "text" } })
    fn parse(input: proc_macro2::TokenStream) -> syn::Result<Self> {
        let mut tokens = input.into_iter();
        
        // Parse the tag name (e.g., div)
        let tag = Ident::parse(&mut tokens)?;
        
        // Expect a block (e.g., { ... })
        let _ = Token![{].parse(&mut tokens)?;
        let block = ExprBlock::parse(&mut tokens)?;
        let _ = Token![}].parse(&mut tokens)?;
        
        // Parse attributes and children from the block
        let (attrs, children) = Self::parse_block_content(block)?;
        
        Ok(HtmlElement { tag, attrs, children })
    }
    
    // Parse attributes (key: value) and children from the block
    fn parse_block_content(block: ExprBlock) -> syn::Result<(Vec<(Ident, LitStr)>, Vec<HtmlNode>)> {
        let mut attrs = Vec::new();
        let mut children = Vec::new();
        
        for stmt in block.stmts {
            if let syn::Stmt::Expr(expr) = stmt {
                match expr {
                    // Attribute: key: "value"
                    Expr::Assign(assign) => {
                        let key = if let Expr::Path(path) = *assign.left {
                            path.path.get_ident().cloned()
                        } else {
                            None
                        }.ok_or_else(|| syn::Error::new_spanned(assign.left, "Expected attribute key"))?;
                        
                        let value = if let Expr::Lit(lit) = *assign.right {
                            if let syn::Lit::Str(lit_str) = lit.lit {
                                lit_str
                            } else {
                                return Err(syn::Error::new_spanned(assign.right, "Expected string literal for attribute value"));
                            }
                        } else {
                            return Err(syn::Error::new_spanned(assign.right, "Expected string literal for attribute value"));
                        };
                        
                        attrs.push((key, value));
                    }
                    // Child element or text
                    _ => {
                        let child = Self::parse_child(expr)?;
                        children.push(child);
                    }
                }
            }
        }
        
        Ok((attrs, children))
    }
    
    // Parse a child node (element or text)
    fn parse_child(expr: Expr) -> syn::Result<HtmlNode> {
        match expr {
            // Text node: "string"
            Expr::Lit(lit) => {
                if let syn::Lit::Str(lit_str) = lit.lit {
                    Ok(HtmlNode::Text(lit_str))
                } else {
                    Err(syn::Error::new_spanned(lit, "Expected string literal for text node"))
                }
            }
            // Child element: tag { ... }
            Expr::Call(call) => {
                let tag = if let Expr::Path(path) = &*call.func {
                    path.path.get_ident().cloned()
                } else {
                    None
                }.ok_or_else(|| syn::Error::new_spanned(call.func, "Expected element tag"))?;
                
                let block = if let Some(arg) = call.args.into_iter().next() {
                    if let Expr::Block(block) = arg {
                        block
                    } else {
                        return Err(syn::Error::new_spanned(arg, "Expected block for element content"));
                    }
                } else {
                    return Err(syn::Error::new_spanned(call, "Expected block for element content"));
                };
                
                let (attrs, children) = Self::parse_block_content(block)?;
                Ok(HtmlNode::Element(HtmlElement { tag, attrs, children }))
            }
            _ => Err(syn::Error::new_spanned(expr, "Invalid child node in html! macro")),
        }
    }
    
    // Generate Rust code to build the HTML string
    fn to_code(&self) -> proc_macro2::TokenStream {
        let tag = &self.tag;
        
        // Generate attribute strings (e.g., class="container")
        let attr_strings = self.attrs.iter().map(|(key, value)| {
            quote! { format!("{}=\"{}\"", stringify!(#key), #value) }
        });
        
        // Generate child code
        let child_code = self.children.iter().map(|child| match child {
            HtmlNode::Element(el) => el.to_code(),
            HtmlNode::Text(text) => quote! { #text.to_string() },
        });
        
        quote! {
            format!(
                "<{} {}>{}</{}>",
                stringify!(#tag),
                vec![#(#attr_strings),*].join(" "),
                vec![#(#child_code),*].join(""),
                stringify!(#tag)
            )
        }
    }
}

#[proc_macro]
pub fn html(input: TokenStream) -> TokenStream {
    let element = parse_html_input(input);
    let code = element.to_code();
    TokenStream::from(code)
}

}

#[proc_macro_derive(Configurable, attributes(config))]
pub fn derive_configurable(input: TokenStream) -> TokenStream {
    let input = parse_macro_input!(input as DeriveInput);
    let type_name = input.ident;
    
    let impl_block = match input.data {
        syn::Data::Struct(data) => match data.fields {
            Fields::Named(fields) => {
                let field_code = fields.named.iter().map(|field| {
                    generate_field_code(field)
                });
                
                quote! {
                    impl config_trait::Configurable for #type_name {
                        fn load() -> Result<Self, config_trait::ConfigError> {
                            Ok(Self {
                                #(#field_code),*
                            })
                        }
                    }
                }
            }
            _ => panic!("Configurable only supports structs with named fields"),
        },
        _ => panic!("Configurable only supports structs"),
    };
    
    TokenStream::from(impl_block)
}

// Generate code for a single struct field
fn generate_field_code(field: &Field) -> proc_macro2::TokenStream {
    let field_name = field.ident.as_ref().unwrap();
    let field_type = &field.ty;
    
    // Extract config attributes (env, default)
    let mut env_var = stringify!(#field_name).to_uppercase();
    let mut default: Option<LitStr> = None;
    
    for attr in &field.attrs {
        if attr.path().is_ident("config") {
            if let Ok(Meta::List(list)) = attr.parse_meta() {
                for nested_meta in list.nested {
                    if let NestedMeta::Meta(Meta::NameValue(nv)) = nested_meta {
                        if nv.path.is_ident("env") {
                            if let Lit::Str(lit_str) = nv.lit {
                                env_var = lit_str.value();
                            }
                        } else if nv.path.is_ident("default") {
                            if let Lit::Str(lit_str) = nv.lit {
                                default = Some(lit_str);
                            }
                        }
                    }
                }
            }
        }
    }
    
    // Generate code to load the env var and parse it
    let env_var_lit = LitStr::new(&env_var, proc_macro2::Span::call_site());
    let parse_code = generate_parse_code(field_type, &env_var_lit);
    
    let load_code = if let Some(default_val) = default {
        // Use default if env var is not set
        quote! {
            std::env::var(#env_var_lit)
                .map_err(|_| config_trait::ConfigError::MissingVar(#env_var_lit.to_string()))
                .and_then(|s| #parse_code)
                .unwrap_or_else(|_| #parse_code.unwrap())
        }
    } else {
        // Require env var to be set
        quote! {
            std::env::var(#env_var_lit)
                .map_err(|_| config_trait::ConfigError::MissingVar(#env_var_lit.to_string()))
                .and_then(|s| #parse_code)?
        }
    };
    
    quote! {
        #field_name: #load_code
    }
}

// Generate code to parse a string into the field type

        }

                        "i32".to_string(),

                    ))
            }
        }
        Type::Path(path) if path.path.is_ident("u32") => {
            quote! {
                s.parse::<u32>()
                    .map_err(|e| config_trait::ConfigError::ParseError(
                        #env_var.to_string(),
                        "u32".to_string(),
                        e.to_string()
                    ))
            }
        }
        Type::Path(path) if path.path.is_ident("bool") => {

6.1 Error Handling

Procedural macros should return clear, actionable error messages to help users debug issues. Use syn::Error to generate compile errors with spans (pointing to the exact location of the issue in the user’s code). Avoid panicking, as it produces unhelpful error messages.

#![allow(unused)]
fn main() {
// Good: Use syn::Error to generate a spanned error
fn validate_field(field: &Field) -> syn::Result<()> {
    if field.ty.is_path() && field.ty.as_path().path.is_ident("Option") {
        return Err(syn::Error::new_spanned(
            field,
            "Option fields are not supported by this macro (yet)",
        ));
    }
    Ok(())
}

// Bad: Panic (produces a generic error)
fn validate_field_bad(field: &Field) {
    if field.ty.is_path() && field.ty.as_path().path.is_ident("Option") {
        panic!("Option fields are not supported");
    }
}

}

6.2 Performance Considerations

Procedural macros run at compile time, so inefficient code can slow down compilation. Optimize by:

  • Minimizing token stream parsing (reuse parsed AST nodes where possible).

  • Avoiding unnecessary allocations (use proc_macro2::Span efficiently).

  • Using syn and quote APIs that minimize overhead (e.g., quote! is optimized for code generation).

Crate serde

Serde is a powerful serialization/deserialization framework for Rust, designed to convert Rust data structures into formats like JSON, BSON, YAML, and more, and vice versa. It stands out for its flexibility, performance, and seamless integration with Rust’s type system. Unlike many other serialization libraries, Serde uses Rust’s procedural macros to generate efficient code at compile time, avoiding runtime overhead and ensuring type safety.

This tutorial is based on Serde v1.0.228 (the latest stable version as of writing), aligned with the official documentation from serde.rs, crates.io, and docs.rs. We will cover core features, advanced usage, and a practical project to solidify your understanding.

Setup and Basic Concepts

1. Setup and Basic Concepts

1.1 Adding Serde to Your Project

To use Serde, add it as a dependency in yourCargo.toml. Serde itself is a core crate, but you’ll need additional crates (called “data formats”) for specific serialization targets (e.g., JSON). The most common format is serde_json.

[dependencies]
serde = { version = "1.0.228", features = ["derive"] }
serde_json = "1.0" # For JSON serialization/deserialization

The derive feature enables procedural macros (Serialize and Deserialize) to automatically implement serialization traits for your data structures.

1.2 Core Traits: Serialize and Deserialize

Serde’s functionality revolves around two key traits:

  • Serialize: Defines how to convert a Rust value into a serialized format. Types implementing this trait can be written to a serializer.

  • Deserialize: Defines how to convert a serialized format back into a Rust value. Types implementing this trait can be read from a deserializer.

For most structs and enums, you can derive these traits with #[derive(Serialize, Deserialize)], which generates the necessary code at compile time.

1.3 Basic Serialization/Deserialization Example

Let’s start with a simple struct representing a user. We’ll serialize it to JSON and then deserialize it back to a Rust value.

use serde::{Serialize, Deserialize};
use serde_json;

// Derive Serialize and Deserialize for User
#[derive(Debug, Serialize, Deserialize)]
struct User {
    id: u32,
    username: String,
    email: String,
    is_active: bool,
    age: Option<u8>, // Optional field (handled gracefully)
}

fn main() {
    // Create a User instance
    let user = User {
        id: 1,
        username: "johndoe".to_string(),
        email: "john@example.com".to_string(),
        is_active: true,
        age: Some(30),
    };

    // Serialize User to JSON string
    let json_string = serde_json::to_string(&user).expect("Failed to serialize user");
    println!("Serialized JSON: {}", json_string);

    // Deserialize JSON string back to User
    let deserialized_user: User = serde_json::from_str(&json_string)
        .expect("Failed to deserialize user");
    println!("Deserialized User: {:?}", deserialized_user);
}

Output:

Serialized JSON: {"id":1,"username":"johndoe","email":"john@example.com","is_active":true,"age":30}
Deserialized User: User { id: 1, username: "johndoe", email: "john@example.com", is_active: true, age: Some(30) }

1.4 Key Notes

  • Optional fields (Option<T>) are serialized as null if None, and omitted in some formats (configurable).

  • Serde handles most Rust primitives (u8, i32, String, Vec, HashMap, etc.) out of the box.

  • Error handling: serde_json::to_string and from_str return Result, so always handle errors (we use expect for brevity here).

Advanced Field Configuration

2. Advanced Field Configuration

Serde allows fine-grained control over how fields are serialized/deserialized using attributes. This is useful for matching external data formats (e.g., JSON with snake_case keys when your Rust struct uses CamelCase).

2.1 Renaming Fields

Use #[serde(rename = "new_name")] to change the name of a field in the serialized output.

#![allow(unused)]
fn main() {
#[derive(Debug, Serialize, Deserialize)]
struct Product {
    product_id: u32,
    #[serde(rename = "product_name")] // Rename to snake_case for JSON
    productName: String,
    #[serde(rename(serialize = "priceUSD", deserialize = "price_usd"))]
    price: f64, // Different names for serialization and deserialization
}
}

2.2 Omitting Fields

  • #[serde(skip)]: Omit the field entirely from serialization/deserialization.

  • #[serde(skip_serializing)]: Omit only during serialization.

  • #[serde(skip_deserializing)]: Omit only during deserialization.

#![allow(unused)]
fn main() {
#[derive(Debug, Serialize, Deserialize)]
struct User {
    id: u32,
    username: String,
    #[serde(skip_serializing)] // Don't send password in serialized output
    password_hash: String,
    #[serde(skip)] // Omit from both serialize and deserialize
    last_login_timestamp: u64,
}
}

2.3 Default Values

Use #[serde(default)] to set a default value if the field is missing during deserialization. The type must implement the Default trait.

#![allow(unused)]
fn main() {
#[derive(Debug, Serialize, Deserialize)]
struct Settings {
    theme: String,
    #[serde(default)] // Use Default::default() if missing
    dark_mode: bool, // Defaults to false
    #[serde(default = "default_font_size")] // Custom default function
    font_size: u8,
}

// Custom default function (must return the same type as the field)
fn default_font_size() -> u8 {
    16
}
}

2.4 Flattening Nested Structs

The #[serde(flatten)] attribute merges the fields of a nested struct into the parent struct’s fields in the serialized output.

#[derive(Debug, Serialize, Deserialize)]
struct Address {
    street: String,
    city: String,
    zip_code: String,
}

#[derive(Debug, Serialize, Deserialize)]
struct User {
    id: u32,
    username: String,
    #[serde(flatten)] // Merge Address fields into User's JSON
    address: Address,
}

fn main() {
    let user = User {
        id: 1,
        username: "johndoe".to_string(),
        address: Address {
            street: "123 Main St".to_string(),
            city: "New York".to_string(),
            zip_code: "10001".to_string(),
        },
    };

    let json = serde_json::to_string_pretty(&user).unwrap();
    println!("{}", json);
}

Output (flattened address fields):

{
  "id": 1,
  "username": "johndoe",
  "street": "123 Main St",
  "city": "New York",
  "zip_code": "10001"
}

Enums and Polymorphism

3. Enums and Polymorphism

Enums are a powerful Rust feature, and Serde provides flexible ways to serialize/deserialize them. By default, enums are serialized as objects with a "type" field (for the variant name) and a "value" field (for the variant’s data).

3.1 Basic Enum Serialization

#[derive(Debug, Serialize, Deserialize)]
enum PaymentMethod {
    CreditCard(String), // Variant with a String (card number)
    PayPal(String),    // Variant with a String (email)
    Cash,              // Unit variant (no data)
}

#[derive(Debug, Serialize, Deserialize)]
struct Payment {
    amount: f64,
    method: PaymentMethod,
}

fn main() {
    let payment = Payment {
        amount: 99.99,
        method: PaymentMethod::CreditCard("4111-1111-1111-1111".to_string()),
    };

    let json = serde_json::to_string_pretty(&payment).unwrap();
    println!("{}", json);
}

Output (default enum serialization):

{
  "amount": 99.99,
  "method": {
    "CreditCard": "4111-1111-1111-1111"
  }
}

3.2 Custom Enum Serialization

Use attributes to change how enums are serialized:

  • #[serde(tag = "type")]: Explicitly set the tag field name.

  • #[serde(untagged)]: Serialize variants without a tag (risky but useful for certain formats).

  • #[serde(rename_all = "snake_case")]: Rename all variants to snake_case.

#![allow(unused)]
fn main() {
#[derive(Debug, Serialize, Deserialize)]
#[serde(tag = "payment_type", rename_all = "snake_case")]
enum PaymentMethod {
    CreditCard(String),
    PayPal(String),
    Cash,
}

// Serialized output will now have "payment_type" instead of variant name as key
// Variants are snake_case: "credit_card", "paypal", "cash"
}

Custom Serialization/Deserialization

4. Custom Serialization/Deserialization

For complex types (e.g., third-party structs without Serialize/Deserialize implementations), you can manually implement the Serialize and Deserialize traits instead of deriving them.

4.1 Manual Serialize Implementation

To implement Serialize, you need to write the type’s data to a Serializer using methods like serialize_struct, serialize_str, etc.

use serde::{Serialize, Serializer};
use std::time::Duration;

// Duration doesn't derive Serialize by default, so we implement it manually
impl Serialize for Duration {
    fn serialize<S>(&self, serializer: S) -> Result<S::Ok, S::Error>
    where
        S: Serializer,
    {
        // Serialize Duration as a string like "5s" (seconds)
        let seconds = self.as_secs();
        serializer.serialize_str(&format!("{}s", seconds))
    }
}

fn main() {
    let duration = Duration::from_secs(5);
    let json = serde_json::to_string(&duration).unwrap();
    println!("Serialized Duration: {}", json); // Output: "5s"
}

4.2 Manual Deserialize Implementation

Implementing Deserialize is more complex, as you need to parse data from a Deserializer. Use Visitor to handle different data types.

use serde::{de::Visitor, Deserialize, Deserializer};
use std::fmt;
use std::time::Duration;

// Implement Deserialize for Duration
impl Deserialize for Duration {
    fn deserialize<D>(deserializer: D) -> Result<Self, D::Error>
    where
        D: Deserializer<'de>,
    {
        deserializer.deserialize_str(DurationVisitor)
    }
}

// Visitor to parse a string like "5s" into Duration
struct DurationVisitor;

impl<'de> Visitor<'de> for DurationVisitor {
    type Value = Duration;

    fn expecting(&self, formatter: &mut fmt::Formatter) -> fmt::Result {
        formatter.write_str("a string in the format 'Xs' where X is a number of seconds")
    }

    fn visit_str<E>(&self, s: &str) -> Result<Self::Value, E>
    where
        E: serde::de::Error,
    {
        // Remove the 's' suffix and parse the number
        let seconds_str = s.strip_suffix('s').ok_or_else(|| {
            E::custom(format!("invalid duration format: {}", s))
        })?;
        let seconds = seconds_str.parse::<u64>().map_err(|e| {
            E::custom(format!("invalid number of seconds: {}", e))
        })?;
        Ok(Duration::from_secs(seconds))
    }
}

fn main() {
    let json = "\"10s\"";
    let duration: Duration = serde_json::from_str(json).unwrap();
    println!("Deserialized Duration: {:?}", duration); // Output: 10s
}

#![allow(unused)]
fn main() {
use serde::de::Deserializer;

// Validate port is between 1 and 65535 during deserialization
fn validate_port<'de, D>(deserializer: D) -> Result<u16, D::Error>
where
    D: Deserializer<'de>,
{
    let port: u16 = u16::deserialize(deserializer)?;
    if port >= 1 && port <= 65535 {
        Ok(port)
    } else {
        Err(serde::de::Error::custom(format!(
            "invalid port: {} (must be 1-65535)",
            port
        )))
    }
}
}
